# Code-page non-bijection matrix (generated)

Generated by `tools/gen_charset_tables.py` from the vendored Unicode
Consortium MICSFT mappings. For each baked code page, this records where
`codepage -> Unicode -> codepage` is **not** the identity, i.e. the carved-out
exceptions to `CodepageRoundTripsOnBijectiveSubset` (`specs/encoding.allium`).

Two kinds of non-bijection:

- **Undefined byte positions** decode to U+FFFD (`codepage_to_unicode` is
  total) and have no reverse, so they do not round-trip.

- **Duplicate rows**: several byte sequences decode to the same code unit
  (vendor-divergent / NEC-IBM / compatibility rows). Only the **canonical**
  sequence (the lowest byte value) re-encodes; the others are the documented
  non-round-tripping exceptions. The device applies the canonical mapping, so
  an inbound path round-trips to the bytes the OS itself uses.

The encode direction is **strict**: a code unit with no byte representation
is rejected (never `'?'`-substituted) - `StrictNarrowingRejectsUnrepresentable`.

## cp858 (derived)

cp858 is **not** in the upstream MICSFT/PC set. It is derived from cp850 by
the single documented override `0xD5` -> `U+20AC` (EURO SIGN), which in cp850
is `U+0131` (LATIN SMALL LETTER DOTLESS I). Every other position is cp850.

## Notable named cases

- **cp932 wave-dash / fullwidth tilde**: `0x8160` decodes to **U+FF5E**
  (FULLWIDTH TILDE), the Microsoft-canonical mapping, *not* JIS U+301C. This
  is the documented vendor-divergent choice; the device applies it as-is.

- **cp932 duplicate rows** (canonical = first; 396 code units affected). Examples:
  - U+2116 <- 0x8782 0xFA59 (canonical 0x8782)
  - U+2121 <- 0x8784 0xFA5A (canonical 0x8784)
  - U+2160 <- 0x8754 0xFA4A (canonical 0x8754)
